Dataflyt — OPC UA gjennom brannmur (steg for steg)
1
OPC DA-simulering (internt i 3p-hvl-app01)
ABB OPC Connect Server + Matrikon genererer tags og datapunkter som simulerer ABB 800xA DCS-strukturen. DCOM-trafikken er lokal på VM-en og krysser aldri brannmuren.
2
OPC DA → OPC UA med UaWrapper v1.9 på app01
UaWrapper v1.9 leser OPC DA-data via lokal DCOM og eksponerer dem som OPC UA-server på opc.tcp://10.0.45.10:46040. Tag-whitelisting i konfigurasjonsfilen (OPCDAClientConfig.xlsx) sikrer at kun ca. 100 av 1000 tags fra 800xA eksponeres videre (SR 7.7).
3
OPC UA gjennom ASA-brannmur — to eksplisitte sesjonsretninger
Når NORØNNA leser tags fra AURORA, initierer NORØNNA-siden (10.0.70.11) TCP-tilkoblingen mot AURORAs OPC UA-endepunkt på port 46040. Når AURORA leser tags fra NORØNNA, initierer AURORA en egen sesjon mot NORØNNAs endepunkt på port 46041. NORONNA_GW_IN tillater kun TCP/46040 fra 10.0.70.11 til 10.0.45.10, og OPC_UA_ZONE_IN kun TCP/46041 fra 10.0.45.10 til 10.0.70.11. Returtrafikk tillates statefully.
4
Mottak og verifikasjon på app02
UaExpert på app02 kobler til UaWrapper-endepunktet (opc.tcp://10.0.45.10:46040) og abonnerer på de whitelistede taggene — tilsvarende verifikasjonssteget i ABBs testprosedyre (3AJG03xx-0001). Verdier simulert i Matrikon på AURORA-siden gjenfinnes i UaExpert på NORØNNA-siden.
5
(Valgfritt) IMS Server aggregerer (VLAN 71)
Aggregating UA Server samler fra Cogent DataHub og eksponerer samlet OPC UA-feed til evt. 3rd party klienter. Tilsvarer ABB IMS Server 87JS000 på site.

Sesjonsinitiering vs. dataretning: Den bidireksjonelle OPC UA-dataflyten består av to eksplisitt definerte sesjonsretninger — ikke en generell åpning mellom sonene. NORØNNA initierer mot AURORA på TCP/46040, AURORA initierer mot NORØNNA på TCP/46041. Tagverdiene returneres som svartrafikk i etablerte TCP-sesjoner (stateful inspection).
